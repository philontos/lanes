# 工程评审准则 (engineering-rubric.md)

review phase 的**质量门判据**。review 是一次**独立评审**(不改代码,只裁判):拿整份
diff 逐条对照下面每一条;**任何一条被违反就判 `reject`** 并写明违反了哪条,否则 `pass`。
判决写进 `<cycle>/verdict.json` —— 严格是 `{"verdict":"pass"|"reject","reasons":[...]}`;
`reject` 会带着 `reasons` 把 impl 阶段重跑去修。

> 这份文件 = 你对"什么算合格的工程实现"的手写基线。和 `judge-principles.md` 一样,
> **由你拥有、自己增删**;judge/review 只是执行它。越具体越好。

---

## 红线:绝不许的 hack(命中任意一条 → reject)
- 不许为了让测试通过而**改弱、删除、跳过测试**(注释掉、`.skip`/`xfail`、放宽断言、缩小覆盖)。
- 不许**为骗过测试而特判/写死**(针对测试输入做特殊分支、硬编码期望输出)。
- 不许**绕过或禁用**校验 / 类型检查 / lint / 构建 / git hook 来"达成"目标。
- 不许**吞错**:空 `catch`、吞掉异常、静默降级、把失败伪装成成功。
- 不许留**假实现 / 占位**却声称完成(`throw NotImplemented`、写死返回值、空函数体)。

## 范围与一致性
- 改动**严格限定在需求(spec)范围内**:不顺手加 feature,不大改无关代码,不做未要求的重构。
- 实现与 `spec.md` / `plan.md` **一致**;spec 的成功标准逐条满足,声称做了的都真做了。

## 遵循成熟最佳实践与既有约定
- 优先采用**成熟、被广泛验证的最佳实践与惯用法**,而非自创或非常规的做法;与工程已有约定(命名 / 目录结构 / 代码风格 / 错误处理)有冲突时,**跟随既有约定**。
- **够用即可、必要时再加**:不要一上来就搭复杂的体系 / 通用框架;先做满足当前需求的**最小且稳定**的实现,**稳定优先**,等确有需要时再扩展(YAGNI)。
- **减少为特异场景定制的代码**:优先用**通用、符合常规**的方案解决问题;尽量不为个别 / 狭窄场景写一次性特判、特例分支或硬编码——确有必要时要在代码或产物里说明理由。
- **依赖选最优解,不为省依赖而 hack**:该用行业标准 / 最佳实践的依赖时就引入(选最主流、维护良好的);**绝不为了不加依赖,而在现有依赖上堆砌 hack / workaround 或重造轮子**。同时也不加冗余、重复、可有可无的依赖。
- 不留**死代码、调试输出**(`console.log`/`print`)、被注释掉的代码块、`TODO` 占位、脚手架残渣。

## 鲁棒性(但不过度设计)
- 对**边界、空值、异常输入、错误路径**要有**明确且一致**的处理;失败要**可观测**(显式报错 / 返回错误),不静默吞掉。
- 关键路径要稳:不因正常的外部波动(空数据、并发、网络 / IO 失败)轻易崩溃。
- 但**不过度设计**:鲁棒 ≠ 堆砌用不上的抽象 / 配置 / 通用化;在"够用、可逆、简单"的前提下**把边界做满**,而不是预防一切。

## 可读与可维护
- 改动**清晰可读**;命名达意;只在"为什么"非显而易见处留简短注释,不加注释噪音。
- 构建与测试**全绿**且未被削弱(若工程有 lint/typecheck,也应通过)。
